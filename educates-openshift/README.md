# educates-openshift

Helm chart that deploys the [Educates](https://educates.dev) training platform
onto **OpenShift Local (CRC)**. Runs online today; goes fully air-gapped by
setting a single value (`global.registry.host`). Ships in three toggled phases —
phase 1 (bare portal) is on by default; phases 2 and 3 are off.

Pinned to **Educates 3.7.2**.

> **Workshops live in a separate chart.** This chart installs the platform and,
> by default (`portal.enabled=true`), a built-in smoke-test portal hosting
> lab-k8s-fundamentals. To manage workshops independently, set
> `portal.enabled=false` here and use the
> [`educates-workshops`](../educates-workshops) chart instead.

---

## How this chart wraps the installer (design note)

Educates is not a native Helm chart — it ships as Carvel `imgpkg` bundles
installed as two packages (`educates-cluster-essentials`,
`educates-training-platform`). The official installer is a **kapp-controller
`App` custom resource** that fetches the `ghcr.io/educates/educates-installer`
imgpkg bundle, templates it with `ytt` using a config secret, resolves images
with `kbld`, and deploys with `kapp`. That single App installs *both* packages.

This chart therefore **renders the official installer App plus its config
secret, RBAC, and the portal/workshop CRs** — it does *not* run a bootstrap Job
that shells out to the `educates` CLI. Rationale: the App approach uses only
upstream, digest-locked official artifacts (no custom installer image to build,
scan, or maintain), it is declarative and reconciling (idempotent under `helm
upgrade` + kapp re-apply), and `kbld` inside the App handles image relocation for
air-gap for free. A CLI-in-a-Job wrapper would add a custom image, imperative
retry logic, and a second copy of the install flow — more moving parts and less
reliable on a single-node CRC. The one cost is a prerequisite: **kapp-controller
must be present** (a one-line apply on CRC, below).

## Prerequisites

- OpenShift Local (CRC) running, logged in as `kubeadmin` (`oc whoami` works).
- `helm` and `oc` on PATH.
- **kapp-controller** installed (reconciles the installer App):

  ```sh
  oc apply -f https://github.com/carvel-dev/kapp-controller/releases/latest/download/release.yaml
  oc rollout status -n kapp-controller deploy/kapp-controller
  ```

## Phase 1 — install the bare platform + portal

```sh
# Render + apply (installer namespace, RBAC, config, App CR, workshops, portal).
helm template educates ./educates-openshift | oc apply -f -
```

The installer App now reconciles (2–5 min on CRC). The `Workshop`/`TrainingPortal`
CRs may error on the first apply because their CRDs are still being installed —
this is expected. **Re-run the same command once the platform is ready** (the
apply is idempotent):

```sh
# Watch the installer App go ReconcileSucceeded.
oc get app installer.educates.dev -n educates-installer -w

# Re-apply so the Workshop/TrainingPortal CRs land now that CRDs exist.
helm template educates ./educates-openshift | oc apply -f -
```

> Prefer `helm install`? It works too, but Helm will fail on the CRs if the CRDs
> aren't present yet. The two-step `helm template | oc apply` (re-applied) is the
> reliable path on a cold cluster.

### Verify

```sh
# Platform pods healthy.
oc get pods -n educates

# Portal is served by the OpenShift Router via the Ingress Educates created.
oc get ingress -n container-paas-ui

# Portal URL (registration is anonymous in phase 1 — just open it):
echo "https://container-paas-ui.apps-crc.testing"
```

Open the URL — the "Container/PaaS Enablement" portal loads and lists the
**Kubernetes Fundamentals** workshop. Click it to start a session (terminal,
console, editor, slides). That confirms cluster-essentials + training-platform +
a session namespace (budget medium, policy restricted) all work.

Portal admin credentials are auto-generated by Educates (never in values):

```sh
oc get secret -n container-paas-ui training-portal -o jsonpath='{.data}'
```

## Phase 2 — front the portal with OpenShift OAuth (`auth.enabled=true`)

Authentication is delegated to OpenShift's **built-in OAuth server** via the
native **`oauth-proxy`** pattern, using **ServiceAccount-as-OAuthClient**: the
proxy's ServiceAccount carries a `serviceaccounts.openshift.io/oauth-redirecturi`
annotation that registers it as an OAuth client with the cluster OAuth server,
and `oauth-proxy` uses the SA's own token as the client secret
(`--openshift-service-account`). Chosen over oauth2-proxy (no OIDC
issuer/discovery wiring) **and** over an explicit `OAuthClient` CR: the SA
variant has **no client secret to template, store in git, or keep in sync**,
which is what makes it safe under ArgoCD (see the GitOps section). A standalone
`oauth-proxy` Deployment sits in front of the Educates portal Service on a
separate host.

```sh
helm template educates ./educates-openshift --set auth.enabled=true | oc apply -f -
```

Renders: the ServiceAccount (with redirect annotation), the `oauth-proxy`
Deployment + Service (OpenShift-issued serving cert) + reencrypt `Route`, the
`system:auth-delegator` binding the proxy needs for token review, and a **cookie**
secret. The cookie secret is generated + preserved via `lookup` for plain
helm/oc; **for ArgoCD, pre-create it and set `auth.existingSecret`** (helm
template has no cluster access, so a generated cookie would churn on every sync):

```sh
oc create secret generic educates-oauth-proxy \
  --from-literal=cookie-secret=$(openssl rand -hex 16) -n educates   # 32 bytes; base64 32 = 44 bytes and fails
# then: --set auth.enabled=true --set auth.existingSecret=educates-oauth-proxy
```

Authenticated portal URL: `https://container-paas-secure.apps-crc.testing`.

Log in with any cluster user (e.g. `kubeadmin`); the proxy forwards to the portal
only after a successful OpenShift login.

> **SCC note:** the proxy runs unprivileged (non-root, port 8443) under the
> default `restricted-v2` SCC — no custom SCC grant is required or created.

## Phase 3 — vcluster per session (`vcluster.enabled=true`)

```sh
helm template educates ./educates-openshift --set vcluster.enabled=true | oc apply -f -
```

Patches the workshop session spec to enable `session.applications.vcluster`,
raises the namespace budget to `large`, and relaxes the session policy from
`restricted` to `baseline` (the vcluster control-plane pod needs it). vcluster
images (`ghcr.io/loft-sh/*`) ship inside the installer bundle, so they are
registry-overridable exactly like everything else (no separate image ref).

> **Extra permissions:** vcluster runs a control plane inside each session
> namespace and syncs a subset of resources to the host. Educates' session
> manager grants the per-session RBAC for this; it is scoped to the session
> namespace and only created when `vcluster.enabled=true`. On OpenShift the
> vcluster pod runs under `baseline` (set above); no cluster-wide grant is added
> by this chart for vcluster.

Combine phases freely, e.g. `--set auth.enabled=true --set vcluster.enabled=true`.

## Offline / air-gapped install

Going offline requires **only** setting `global.registry.host` (plus mirroring
the images below). Every image reference in the chart rewrites to that host, and
Educates' own `imageRegistry.host` is set so its components pull from the mirror.

### 1. Mirror the images

The installer bundle carries **every Educates platform image** (controllers,
session images, docker-registry, vcluster images, etc.). Relocate the whole
bundle with `imgpkg` — this copies all of them in one shot:

```sh
imgpkg copy \
  -b ghcr.io/educates/educates-installer:3.7.2 \
  --to-repo YOUR_REGISTRY/educates/educates-installer
```

Then mirror each workshop content image (one per workshop):

```sh
skopeo copy \
  docker://ghcr.io/educates/lab-k8s-fundamentals-files:8.1 \
  docker://YOUR_REGISTRY/educates/lab-k8s-fundamentals-files:8.1
```

For phase 2, also mirror the oauth-proxy image
(`registry.redhat.io/openshift4/ose-oauth-proxy:v4.14`).

**Images that must exist in the mirror** (all except the workshop-content and
oauth-proxy images are pulled in automatically by the `imgpkg copy` above; listed
for audit):

### 2. Create a pull secret (if the mirror needs auth)

```sh
oc create ns educates-installer
oc create secret docker-registry mirror-pull \
  --docker-server=YOUR_REGISTRY --docker-username=... --docker-password=... \
  -n educates-installer
```

### 3. Install pointing at the mirror

```sh
helm template educates ./educates-openshift \
  --set global.registry.host=YOUR_REGISTRY \
  --set global.registry.pullSecret=mirror-pull \
  | oc apply -f -
```

Nothing else changes — the App CR pulls the relocated bundle, `kbld` resolves all
platform images from it, and workshop/vcluster/proxy images resolve to the mirror.

## Install via ArgoCD (GitOps)

The chart is ArgoCD-ready. Point an `Application` at it:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: educates-openshift
  namespace: openshift-gitops
spec:
  project: default
  source:
    repoURL: https://your.git/host/repo.git
    targetRevision: main
    path: educates-openshift
    helm:
      values: |
        educates:
          ingressDomain: apps-crc.testing
        # air-gap: global: { registry: { host: YOUR_REGISTRY } }
  destination:
    server: https://kubernetes.default.svc
    namespace: educates        # release namespace (used by phase 2 objects)
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    retry:
      limit: 20                 # CRs retry until the installer creates their CRDs
      backoff: { duration: 15s, maxDuration: 5m, factor: 2 }
```

**Why it works** — three things were designed for this:

1. **Sync waves** order the apply: namespace/RBAC/config (waves -2…-1) → installer
   `App` (wave 0) → `Workshop`/`TrainingPortal`/`Route` (wave 5).
2. The CRs carry `SkipDryRunOnMissingResource=true`, so ArgoCD does not hard-fail
   dry-run before the `training.educates.dev` CRDs exist — combined with sync
   retry, it converges once the installer `App` finishes (no manual re-apply).
3. **Layered reconcilers, cleanly separated:** ArgoCD manages only the `App` CR,
   config, and portal/workshop CRs. kapp-controller owns the hundreds of platform
   resources — invisible to ArgoCD, so **no diff churn** on platform internals.

**Two requirements:**

- **kapp-controller must be present** (the prereq above). ArgoCD does not replace
  it; it deploys the `App` that kapp-controller reconciles. ArgoCD ships a health
  check for `kappctrl.k14s.io/App`, so the Application reports Progressing until
  the platform is healthy. (If your ArgoCD lacks it, add a
  `resource.customizations` health check for that group/kind.)
- **Phase 2 secret:** set `auth.existingSecret` to a pre-created cookie secret.
  `helm template` under ArgoCD has no cluster access, so the `lookup`-based
  generation would produce a new cookie every sync. Phase 1 has no such secret
  and needs nothing extra.

## Security grants introduced by this chart

| Kind | Name | Scope | Reason |
|------|------|-------|--------|
| ClusterRoleBinding | `educates-installer` → `cluster-admin` | cluster | The Educates installer creates CRDs, cluster packages, operators, and SCCs — it requires cluster-admin. Mirrors upstream `educates-installer-app-rbac.yaml`. Phase 1. |
| ServiceAccount | `educates-installer` | `educates-installer` ns | Identity kapp-controller runs the installer App as. Phase 1. |
| ClusterRoleBinding | `educates-oauth-proxy-auth-delegator` → `system:auth-delegator` | cluster | oauth-proxy validates OpenShift tokens via `TokenReview`/`SubjectAccessReview`. Phase 2 only (`auth.enabled`). |
| ServiceAccount | `educates-oauth-proxy` (+ `oauth-redirecturi` annotation) | release ns | Identity the oauth-proxy runs as; its redirect annotation registers it as the OAuth client (no separate `OAuthClient` CR / client secret). Phase 2 only. |

**SCC grants:** none created by this chart. The installer configures Educates to
use `security-context-constraints` as its policy engine and manages workshop SCCs
internally; the oauth-proxy runs under the built-in `restricted-v2` SCC. No
`anyuid`/`privileged` grant is added.

> Educates' own installer additionally creates SCCs and RBAC *inside* the
> platform for running workshop sessions — those are owned by the Educates
> packages, not by this chart, and are governed by
> `clusterSecurity.policyEngine: security-context-constraints`.

## Values reference

See [`values.yaml`](values.yaml) — every option is commented. Highlights:

| Value | Default | Purpose |
|-------|---------|---------|
| `global.registry.host` | `""` | Empty = upstream. Set = rewrite **all** images to this host (air-gap). |
| `global.registry.pullSecret` | `""` | Name of an existing pull secret for the mirror. |
| `educates.version` | `3.7.2` | Educates release / bundle tag. |
| `educates.ingressDomain` | `apps-crc.testing` | CRC wildcard domain. |
| `educates.security.policyEngine` | `security-context-constraints` | Forced on OpenShift. |
| `educates.security.rulesEngine` | `none` | Kyverno off on OpenShift. |
| `portal.title` | `Container/PaaS Enablement` | Portal display title. |
| `portal.registration.enabled` | `true` | Anonymous access (phase 1). |
| `workshops[]` | lab-k8s-fundamentals | Workshops + registry-overridable content image. |
| `auth.enabled` | `false` | Phase 2 — OpenShift OAuth front. |
| `vcluster.enabled` | `false` | Phase 3 — per-session vcluster. |

## Uninstall

```sh
helm template educates ./educates-openshift | oc delete -f - --ignore-not-found
# Educates' own resources are removed by deleting the installer App:
oc delete app installer.educates.dev -n educates-installer
```
