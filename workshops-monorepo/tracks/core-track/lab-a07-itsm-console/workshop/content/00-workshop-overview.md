You've spent five labs doing things yourself with `oc` — deploying, configuring, exposing,
persisting. But on **{{< param product_name >}}** you can't just click everything. Some
actions are **requests**: they go through the platform team via an ITSM ticket. This lab
draws that line and shows you where requests happen.

{{< note >}}
**First time in one of these labs?** See the
[DCS Academy environment guide]({{< param dcs_docs_base_url >}}/academy/environment-guide).
{{< /note >}}

## What You'll Learn

By the end of this workshop you will be able to:

- Explain that {{< param product_short >}} self-service runs through **ITSM requests**, and where the console is.
- Tell which actions need a ticket vs which are self-service via `oc`.
- Describe the **request → approval → provisioning** loop from a walked example.

Requests are documented in the
[{{< param product_short >}} getting-started guide]({{< param dcs_docs_base_url >}}/getting-started/requests).

## Prerequisites

- **A01 — What is DCS?** You've also met the *edges* of self-service in A04 (Routes need
  a PROD namespace) and A05 (S3 comes via a ticket) — this lab names the mechanism behind
  those asides.

## Time and Difficulty

- **Estimated time:** 20 minutes
- **Difficulty:** Beginner
